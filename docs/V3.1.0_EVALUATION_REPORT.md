## 2. REMAINING VULNERABILITIES

### 2.1 New vulnerabilities introduced by v3.1.0: **None found**

The v3.1.0 changes are conservative and well-scoped. No new attack surface was introduced.

### 2.2 Remaining unbounded data structures

Several plain dicts in `ddos_controller.py.__init__` are technically unbounded:

| Variable | Line | Risk | Severity |
|----------|------|------|----------|
| `self.datapaths` | 316 | Bounded by physical switch count | Negligible |
| `self._flow_table_sizes` | 354 | Bounded by switch count | Negligible |
| `self._network_dst_flow_counts` | 357 | Rebuilt each polling cycle | Low |
| `self._network_dst_source_sets` | 358 | Rebuilt each cycle; inner sets unbounded per-dst | Low-Medium |
| `self._pending_stats_replies` | 359 | Bounded by switch count | Negligible |
| `PacketInRateLimiter._counters` | 169 | Bounded by switch count | Negligible |
| `FloodRateLimiter._counters` | 198 | Bounded by switch count | Negligible |

**Verdict:** `_network_dst_source_sets` is the only concern. Each destination IP accumulates a `set()` of source IPs that could theoretically grow large during a spoofed-source DDoS with millions of unique sources. However, it is rebuilt from scratch each 3-second polling cycle, so the growth is bounded by the number of flows in a single stats reply. This is acceptable.

### 2.3 Race conditions in ML pipeline

The producer-consumer pattern uses `eventlet.queue.LightQueue` with `put_nowait()` / `get(timeout=1)`, which is eventlet-safe. The `_aggregate_and_enqueue` method runs in the monitor thread; `_ml_inference_loop` runs in its own green thread. Shared state (`_network_dst_source_sets`, `_network_dst_flow_counts`) is written by the monitor thread and read by `_handle_detection_results` in the ML thread — these are not lock-protected, creating a potential **read-during-write** race. However, in eventlet's cooperative threading model, context switches only happen at I/O points, and the dict assignment at line 1103-1104 is atomic from Python's perspective.

**Verdict:** Low risk under eventlet's cooperative model. Would be a real race condition under preemptive threading.

### 2.4 Unsafe deserialization

`joblib.load()` at `ddos_controller.py:381-382` remains the only deserialization point. It is gated by HMAC-SHA256 verification (`_verify_model_integrity`). This is the expected design. No new deserialization paths were introduced.

### 2.5 Hardcoded secrets: **None found**

All secrets are environment-variable-based: `SDN_API_TOKEN`, `SDN_MODEL_HMAC_KEY`. No hardcoded tokens, passwords, or keys in source.

### 2.6 Input validation gaps: **None found**

New integration tests and adapter tests use synthetic data, not user input. The `_sanitize_ip()` function validates all IPs before logging. `ipaddress.IPv4Address()` validation occurs before block rule installation.

### 2.7 Path/command injection: **None found**

No `os.system()`, `subprocess.run(shell=True)`, or f-string path construction from user input. Log file paths are constructed from `self.log_dir` which is set at init time.

### 2.8 REST API authentication

Bearer token auth in `api_auth.py` uses `hmac.compare_digest()` for timing-safe comparison. Falls back to localhost-only binding when no token is configured. This is correctly implemented.

### Vulnerability Score: **1 found** (down from 6)

The sole remaining concern is the theoretical read-during-write race on `_network_dst_source_sets` between the monitor and ML threads, which is effectively mitigated by eventlet's cooperative scheduling.

## 7. SDN ARCHITECTURE

### Anti-spoofing flow table design
**Correct.** Two-priority scheme: ALLOW at priority 50 with `OFPP_NORMAL` for IPv4, DROP at priority 40 for everything else. Legitimate traffic stays in the data plane. ARP goes to controller for proxy. Per-port per-IP bindings defined in `topology_config.py:68-85`.

### ML decoupling
**Properly decoupled.** ML inference runs in `_ml_inference_loop` green thread, reading from `eventlet.queue.LightQueue`. Stats collection in `_monitor_loop` enqueues batches via `put_nowait()`. No blocking path from ML to event loop. Timeout guard at `ddos_controller.py:1192`.

### ECMP
Group tables correctly configured as `OFPGT_SELECT` with equal weights on spine uplinks. Broadcast groups use `OFPGT_ALL`. Installed on leaf switches only (`ddos_controller.py:709-711`).

### Network-wide blocking
Block rules pushed to ALL switches via `_install_block_on_all_datapaths()`. Leaf switches first, then spine (line 1336). Uses `BLOCK_COOKIE` (0xDEAD) for source-based and `BLOCK_DST_COOKIE` (0xBEEF) for destination-based blocking.

### Stats aggregation
Counts traffic only at leaf switches (`is_leaf` check at line 1033) to prevent double-counting flows traversing multiple switches via ECMP. Network-wide aggregates rebuilt each cycle.

### ARP proxy cache
**Bounded.** `TTLCache(maxsize=1024, ttl=300)` at line 331. Stale entries auto-expire.

### Priority structure
Clear hierarchy: BLOCK=100 > ANTI_SPOOF_ALLOW=50 > ANTI_SPOOF_DROP=40 > FORWARDING=10 > ARP_PROXY=5 > TABLE_MISS=0. No conflicts or shadowing detected.

### Flow table capacity
Monitoring via `_flow_table_monitor` thread with `FLOW_TABLE_WARNING_PCT=0.80` and `FLOW_TABLE_CRITICAL_PCT=0.95`. Emergency eviction deletes forwarding flows, preserving block rules.

### Flood suppression
Two-level: per-switch rate limit (MAC-independent, line 652) + per-flow dedup (line 656-664). Prevents MAC rotation bypass.

### SDN Architecture Score: **8/10**

**Justification:** Sound design with proper data plane / control plane separation. Anti-spoofing is now correct. ECMP is well-implemented. Deductions: (-0.5) `_unblock_ip` doesn't remove BLOCK_DST_COOKIE rules, (-0.5) no flow table eviction by priority ordering (just deletes all forwarding), (-0.5) `_network_dst_source_sets` not lock-protected between threads, (-0.5) no validation that OVS actually supports SELECT group type (assumes hash selection).

### F. IS A v3.2.0 NEEDED?

**No.** The project has reached a stable, release-quality state for its intended purpose as a research prototype and educational tool.

The remaining weaknesses are minor:
- The untested `_aggregate_and_enqueue()` is a test coverage gap, not a bug.
- The integration test approach is pragmatic given Ryu's runtime requirements.
- The docstring inaccuracies and missing git tags are cosmetic.
- The `_unblock_ip()` limitation is an edge case that could be fixed in a point release if needed.

A v3.2.0 would only be justified if:
1. The project needs to support a new dataset or deployment target.
2. A bug is discovered in the detection pipeline.
3. The project transitions from research to production use (which would require far more than a patch).

**Recommendation:** Tag v3.1.0 in git (`git tag -a v3.1.0 -m "Stable release"`), archive the audit trail, and consider the core codebase complete.
